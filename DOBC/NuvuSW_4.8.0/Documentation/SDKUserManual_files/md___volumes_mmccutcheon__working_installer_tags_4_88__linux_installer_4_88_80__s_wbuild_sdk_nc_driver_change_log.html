Nüvü Camēras Software Change Log

Changes for 4.8.0

2022-03-24

SDK changes

  • Added API functions to report the required frame grabber configuration for external image acquisition
  • Added API functions to override frame grabber size restrictions for external image acquisition

Changes for 4.7.0

2021-11-02

NuPixel changes

  • Handles TDI vertical (Y) binning.
  • Dynamically handles binning modes that are unavilable due to ROI settings.

SDK changes

  • TDI line rate explicitly available in API via ncCamGetTdiLinerateNominal(); image header TDI_RATE.
  • TDI vertical (Y) binning enabled. ROI Y-direction upper limit (Y-offset + height) must be a multiple of the binning; input and output ROI regions align at the Y-direction upper limit in the event of any truncation due to partial binning from the choice of Y-direction lower limit.
  • ncCamParamAvailable() can now be used to verify the current availability of a specific binning mode: it will return NC_ERROR_CAM_FEATURE_DISABLED if the binning is not presently compatible but otherwise available.
  • Documentation of enum Features and associated enums.
  • Improved cross-referencing and clarity of documentation.
  • Bugfix for Photon-Counting (PC) processed images identified as LM (bias-subtracted).
  • Bugfix for detecting cameras that use two cables (e.g. CameraLink Extended/Deca for HT4096): sometimes detection would hang due to unexpected serial transmission on Channel 1.

Changes for 4.6.5

2021-09-14

SDK changes

  • Bug fix for logging SDK function calls with pointer arguments that point to non-null terminated buffers

Changes for 4.6.4

  • change_log.txt updated for tags since 4.6.1

Changes for 4.6.3

Changes for 4.6.2

Changes for 4.6.1

2020-10-08

SDK changes

  • Support for serial communication with Pasadena controller using CRC-32 error-checking.
  • New API functions relating to serial communications using different serial protocols : ncGrabFormatSerialCommand(), ncGrabSendSerialCommand().

Changes for 4.5.1

2020-08-19

NuPixel changes

  • Added a Raw Frame panel to display all returned image data.

SDK changes

  • New API functions to provide access to functionality and operational parameters of Fast Kinetics Mode for compatible cameras : ncCamSetFastKineticsMode(), ncCamGetFastKineticsMode(), ncCamSetFastKineticsModeDiscard(), ncCamGetFastKineticsModeDiscard(), ncCamGetFastKineticsModeMaxImages(), ncCamGetFastKineticsModeImageCount(), ncCamGetFastKineticsModeImageParams(). NOTE: Fast Kinetics Mode is not compatible with the use of ShutterMode AUTO (see ncCamSetShutterMode()).
  • Corrected calculation of time periods provided by : ncCamGetIntegrationTime(), ncCamGetFrameTransferDuration(); adapted same for use with Fast Kinetics Mode.
  • New API functions to provide access to image timestamps via ImageParams objects (refer to ncCamOpenImageParams() and ncGrabOpenImageParams()) : ncCamGetImageCtrlTimestamp(), ncCamGetImageHostTimestamp(), ncCamGetImageTimestampMode(), ncGrabGetImageCtrlTimestamp(), ncGrabGetImageHostTimestamp(), ncGrabGetImageTimestampMode().

Changes for 4.4.2

2020-04-13

SDK changes

  • New API functions to select the bias clamp level (the stable, zero-signal reference level used in bias-subtracted images) : ncCamSetBiasClampLevel(), ncCamGetBiasClampLevel(), ncGrabSetBiasClampLevel(), ncGrabGetBiasClampLevel().
  • Bug fix for use of X-binning for fast readout modes.
  • Bug fix for protection against re-opening framegrabber channels that are already in use.
  • Bug fix for compressing images saved individually.
  • Re-work of internal handling of image headers for better consistency.

Changes for 4.4.1

2020-01-30

SDK changes

  • Overhauled HTML documentation: improved accuracy, completeness and navigation.
  • Refined implementation of buffering of raw frames (to safe-guard to the maximum extent possible all acquired data) with the option of using image buffers that are backed by system memory or disk.
  • Extended implementation of disk-backed buffering of raw frames to Linux.
  • Bug fix for internal handling of frame latency: the tagging of images with image metadata (ImageParams) did not correctly account for frame latency.
  • New API functions to verify if an image pointer returned by a ncCamRead() or ncGrabRead() function is still valid and how much valid pixel data it points to: ncCamValidateReadoutImageSize() & ncGrabValidateReadoutImageSize().
  • Removal of deprecated API functions whose nomenclature clashed with that of some newer API functions. The removed functions already have preferred replacements, e.g. ncGrabGetControllerAvailable() should be replaced with ncControllerListGetSize().
  • Made reciprocal gain (k-gain [e^-/ADU]) available in API (ncCamGetKGain()) and image headers (KGAIN_##); up to 16 readout taps may have defined values.
  • Extended available timestamp precision to nanoseconds.
  • Re-introduced compatibility with legacy camera controllers.
  • Improvements to image processing pipeline for increased efficiency and image accounting (i.e. counting overruns, dropped images, etc.); adjustments to default memory allocation for image buffers.
  • Corrections to image headers.

Changes for 4.4.0

2018-12-14

SDK changes

  • Handling of acquisition via various CameraLink modes and pixel depths and via multiple sensor readout taps (requires a compatible framegrabber and framegrabber plugin); in the case that image data must be returned in a compressed form, decompression of raw frames to unprocessed images is handled transparently.
  • The API function ncGrabSetSize() now automatically compensates for the size restrictions imposed by the framegrabber in use, in line with previous documentation, but also requires that the requested image size equal a multiple of the number of active readout taps. In case adjustments to the requested size are necessary, users may use the new API function ncGrabValidateSize() to report what corrections are required.
  • New API functions to help manage the number of allocated image buffers.
  • Overhaul of overrun detection.
  • First implementation of disk-backed buffering of raw frames under Windows.
  • New API function allowing the user to set the camera to an 'idle' state, with no readout mode loaded (similar to when it is first turned on): ncCamClearReadoutMode().
  • New API functions allowing the logging of specified API functions to be suppressed, to help reduce logfile bloat in certain applications: ncCamSilenceFunctionLogging() & ncGrabSilenceFunctionLogging()
  • Fixed a bug relating to the use of an external shutter and controlling the polarity of the Shutter, Arm and Fire external signals.

Changes for 4.3.1

2018-10-25

SDK changes

  • API functions that read images from the acquisiton context now require pointers to const-pointers to the image data (the definition of NcImage has changed). This is necessary to ensure simultaneous saving and access to images without race conditions or synchronisation issues. A new API function allows the user to wait for safe write access to the image buffer, if required: ncCamWaitUntilImageWritable().
  • New API function to block until the save queue has been flushed; the acquisition context waits for the save queue to be flushed fully before closing.
  • It is now safe to call any API functions from callbacks.
  • Handling of TDI readout modes: note that the availability of trigger, timestamp and processing modes and that the definition of exposure and readout time differs between standard (frame transfer or full frame) and TDI readout.
  • Identification of TDI readout modes via new API function allowing the recovery of readout mode flags: ncCamGetReadoutModeFlags()
  • Additional members of the Features enum allow querying for the support of particular processing modes; ncCamSetProcType() will verify support and return an error in the event there is not; when setting a readout mode, support for the current ProcType is verified and, if it is not available, the ProcType returns to NO_PROC.
  • New API function reporting the integration time period over which a pixel should receive light from an observed area of the field of view during the acquisition of an image.
  • New API functions reporting the nominal framerate and maximum framerate available in the current readout configuration: ncCamGetFramerateNominal() and ncCamGetFramerateMaximum(). Useful for determining a reasonable value to pass to ncCamSetTimeout().

Changes for 4.2.1

2018-03-27

NuPixel changes

  • Added a new live image transformation panel to reorient the live image to account for various camera mounting positions.
  • Improved how NuPixel handles controller generated warnings and errors.
  • Improved how NuPixel handles image processing errors.
  • Added the analog offset property to the analog settings panel.
  • Added a menu item to export the image processing bias image.
  • Eliminated redundant information from the log file to reduce disk space usage and improve clarity.
  • Various minor stability and user experience improvements.

SDK changes

  • New API functions provide information that can be used to configure a separate acquisition interface to receive images while the SDK retains control of the camera. This provides the option of a user implementing their own acquisition tools but retaining access to the functionalities of our SDK.
  • New API functions to export the image processing bias.
  • New API function to control which API functions are logged.
  • New API function to automatically and safely warm up the camera before shutdown.
  • New API function to check if an acquisition is currently ongoing.
  • New API function to get the frame transfer duration as an estimate of the dead-time encountered when operating at maximum framerate.
  • New API function to retrieve the model of the connected camera.
  • New API functions to allow readout of extended dark-reference regions from the sensor (available when compiling with NC_ADVANCED defined).
  • New SDK synchronization to support multi-threaded use.
  • Improved the bias image calculation fidelity.
  • Improved the image processing pipeline to better exploit multiple processors.
  • Improved the SDK's camera diagnostic capabilities.
  • Image headers now always include the detector temperature at the time of acquisition.
  • Various minor stability and user experience improvements.
