Detailed Description

These functions allow the set-up of a Nüvü Camēras device (i.e. the configuration of the device for readout) via a NcCam handle.

Devices provided by Nüvü Camēras with pre-programmed readout modes, such as the HNü or EMN2 models of camera, will support the use of NcCam as the control context and it is recommended for most use cases.

Enumeration Type Documentation

enum Ampli

Values identifying possible readout paths that may be provided by a readout mode.

For use with the functions ncCamGetAmpliTypeAvail() and ncCamGetFreqAvail().

Enumerator
NOTHING 

Generic unsupported value.

EM 

Photo-electrons pass through an Electron Multiplication register before readout.

CONV 

Photo-electrons pass directly to the readout register.

enum CommType

Values identifying the type of communication interface that can be used to operate a Nüvü Camēras device.

This typically corresponds to a particular type/manufacturer of framegrabber and support requires that an appropriate plugin be installed.

Enumerator
INVALID 

Fallback value for error states.

AUTO_COMM 

Detect any plugin; equal to (NC_AUTO_UNIT - NC_AUTO_CHANNEL).

VIRTUAL 

Support provided by the nc_grab_virtual plugin. Allows the use of this SDK without having a camera available on your system.

EDT_CL 

Support provided by the nc_grab_edt plugin. For EDT framegrabbers.

PT_GIGE 

Support provided by the nc_grab_pleora plugin. For Pleora GigE Vision framegrabbers and IP Engines providing connectivity over Ethernet.

MIL_CL 

Support provided by the nc_grab_mil plugin. For Matrox framegrabbers.

SAPERA_CL 

Support provided by the nc_grab_sapera plugin. For Teledyne Dalsa framegrabbers.

UNKNOWN 

Default for a user-implemented plugin.

REMOTE_COMM 

Not yet implemented. To allow remote access over Internet.

NC_SERIAL_IO 

Support provided by the nc_grab_serial plugin. No images may be returned but the SDK can be used to configure the camera while a user's own, independent framegrabber implementation receives data.

enum NcPortUnusedReason

Values identifying possible reasons that a communication interface (CommType) might not provide access to a Nüvü Camēras device when this SDK scans your system for an available device.

Enumerator

The port was found to not be connected to a Nuvu device.

NC_PORT_UNUSED_ALREADY_IN_USE 

The device is already in use by another instance.

NC_PORT_UNUSED_FILTERED 

Port was not scanned (filtered, internal use only)

NC_PORT_CLMODE_UNAVAILABLE 

The CameraLink mode required by the device is not available in the current configuration.

NC_PORT_CLMODE_SUPPORT 

The CameraLink mode required by the device is not supported by the plugin used to access it.

NC_PORT_UNUSED_UNKNOWN 

The reason was unusual and has no unique identifier.

int ncCamOpen ( int  unit,
int  channel,
int  nbrBuffer,
NcCam *  cam 
)

Opens a channel from the host system to the camera and initializes internal variables.

Initialises a NcCam object in preparation for controlling the associated camera via calls to ncCam functions: see Camera Set-up Functions (NcCam) and Camera Parameters Functions (NcCam) for a summary. The user may specify a camera via a particular framegrabber unit and channel or use special unit and channel values (see below) to allow automatic discovery of an available camera.

The number of loop buffers required to pipeline image acquisition is also set.

Parameters
unitThis parameter represents the framegrabber type (CommType) and index on which the channel is opened. If only one framegrabber device is to be used, this value should be set to the framegrabber type. Typical supported framegrabber types are: VIRTUAL, EDT_CL, PT_GIGE, MIL_CL and SAPERA_CL. Each framegrabber type is associated with a software plugin which must have been installed correctly in order to use that type of interface. The framegrabber type VIRTUAL allows the use of the SDK without having a camera available on your system; once you have installed the plugin providing the virtual camera, please refer to the READ_ME file in the nc_grab_virtual directory. In addition, NC_AUTO_UNIT is an allowed value of this parameter that will automatically detect a unit on which a camera is available. We recommend using it only when a single camera from Nüvü is available to your system (including any VIRTUAL units) since we cannot guarantee the order in which they will be detected. If your system has multiple framegrabber devices of the same type, simply add the device index you wish to use. Note that the device index starts at 0 (e.g. EDT_CL+1 will access the second EDT_CL board) and that the ordering depends upon the exact configuration of your system. However, if you are using PT_GIGE with multiple devices, the order of units is not guaranteed: you may instead want to use ncCamOpenFromList() after having chosen a device by its serial number (see ncControllerListGetSerial()).
channelThis parameter represents the channel index that is opened. If the first channel is used, this parameter must be set to 0; if the second channel is used, then this parameter should be set to 1; etc. In addition, NC_AUTO_CHANNEL is an allowed value of this parameter that will automatically detect a channel on which a camera is available. We recommend using it only when a single camera from Nüvü is available on the specified unit since we cannot guarantee the order in which they will be detected.
nbrBufferThis parameter represents the number of buffers that will receive acquired images from the camera. These buffers will receive images one after the other such that one or more buffers can be used/processed while another one receives a new image from the camera. Specify a negative value to cause the SDK to estimate an appropriate number of image buffers for the camera's sensor format (recommended).
*camThis parameter is a pointer to the camera structure being opened.
Returns
This function returns "0" if successful and an error code if not.

int ncCamPrepareShutdown ( NcCam  cam,
int  blockingMode,
int(*)(double, double)  callback 
)

Prepares the connected camera to be safely shut down. HNu cameras must be warmed up to a safe temperature in a controlled fashion. This function can be used to preform this warming up. Additionally, the shutter is closed, the EM gain is reset to 1 and any ongoing acquisition is stopped.

Parameters
camThis parameter represents the camera structure to associated with the camera that is being prepared to shutdown. This structure must have been previously defined using the ncCamOpen() function.
blockingModeIf this paramter is 0, the function will configure the camera to prepare it for a shutdown but will return immediatly. If this parameter is not 0 the function will only return once the camera has warmed to a safe temperature or if the provided callback is used to interupt the wait. If the camera is already at a safe temperature or if the camera model does not need to be warmed before shutdown, the blockingMode parameter has no effect.
callbackThis optional callback can be used when the blockingMode parameter is not 0. While waiting for the camera to warm up, the provided callback will be called continuously. The callback's first argument is the current detector temperature and the second argument is an indicator of approximation of the process' progress. The progress value ranges from 0.0 (just started) to 1.0 (just finished).
Returns
This function returns "0" if successful and an error code if not.

int ncCamSetHeartbeat ( NcCam  cam,
int  timeMs 
)

Sets the time interval at which the camera is pinged by the NIC.

By the end of the mentioned time interval, if the camera still has not answered, the communication port will be closed (which usually means the application must be restarted). In debug mode, when on a breakpoint, the camera won't be pinged unless the executed line is changed. Therefore, when using breakpoints, the heartbeat time should be increased.

Note
This function only applies when using a GigE Vision-Ethernet camera.
Parameters
camThis parameter represents the camera structure used to acquire the images. This structure must have been previously defined using the ncCamOpen() function.
timeMsThis parameter represents the time interval at which the camera is being pinged. The default value is 5000ms. The minimum value that can be set is 2000ms.
Returns
This function returns "0" if successful and an error code if not.

Sets a callback function which will be triggered once the camera is allowed to shutdown.

Once this function is called, the temperature will start moving towards -60 degree Celsius. Once the temperature is -60 or above the callback will be triggered since the camera can be safely shutdown.

int ncCamSetReadoutMode ( NcCam  cam,
int  number 
)

Sets the camera's readout mode. The selected readout mode dictates the type of amplifier as well as the horizontal and vertical frequencies used.

When a new readout mode is put in place, the missed images, the overruns, the timeouts, the EM gain, the analog gain, the analof offset, the ROI, the binning and the crop-mode are reset. The exposure time, the waiting time, the various signal polarities, the shutter mode and the trigger mode are kept as before.

Note
This function cannot be performed while an acquisition is in progress (or in queue). To do so, the ncCamAbort function can be used to ensure that there are no pending acquisitions. The acquisition queue will be emptied (you can't retrieve the images previously acquired).
Parameters
camThis parameter represents the camera structure used to acquire the images. This structure must have been previously defined using the ncCamOpen() function.
numberThis parameter indicates the readout mode to be loaded onto the camera. The function ncCamGetNbrReadoutModes can be used to retrieve the total number of modes available on the camera.
Returns
This function returns "0" if successful and an error code if not.

int ncCamGetFramerateNominal ( NcCam  cam,
int  cameraRequest,
double *  fps 
)

Reports the nominal number of frames that should be acquired per second, based on the settings in place on the camera.

See also ncCamGetFramerate(). Reports the frame-rate that may be expected from the camera's current configuration when using the INTERNAL trigger mode (this is the default mode of operation, see ncCamSetTriggermode()). The rate at which frames are returned from the camera depends directly upon the exposure and waiting time requested (see ncCamSetExposureTime() and ncCamSetWaitingTime()) and upon the readout time (see ncCamGetReadoutTime()) which itself depends upon the readout mode and the layout of the image requested (e.g. width, height and position of ROIs, binning). These factors are taken into account in calculating the value reported by this function.

Note
The value recovered from this function may be useful in setting the acquisition timeout (see ncCamSetTimeout()), if the INTERNAL trigger mode is in use: a good general rule is to use a timeout of either twice the inverse of the frame-rate or the inverse of the frame-rate plus one second whichever is smaller. Given that the value reported by this function is in Frames Per Second whilst the timeout is to be set in milliseconds, there is a factor of 1000 between the inverse of this value and that required to calculate the timeout.
Parameters
camThis parameter represents the camera structure used to acquire the images. This structure must have been previously defined using the ncCamOpen() function.
cameraRequestThis parameter is a flag that, when set to a non-zero value, will request up-to-date information directly from the camera. This process is not immediate, as communication through the serial link is required to retrieve the information. If this flag is set to 0, then the camera settings previously set within the application will be used. It is recommended to only set this flag to 1 when the ncCamSetExposureTime() or ncCamSetWaitingTime() functions have not previously been used in the application or when there is a possibility that the camera settings have been changed using another application than the one currently running.
*fpsUpon return, this pointer will contain the nominal frame-rate (in frames per second) according to the camera settings in place.
Returns
This function returns "0" if successful and an error code if not.
int ncCamGetTdiLinerateNominal ( NcCam  cam,
int  cameraRequest,
double *  lineRate_kHz 
)

For TDI readout, reports the nominal rate at which lines are read-out, based on the settings in place on the camera.

Reports the line-rate that may be expected from the camera's current configuration when using the INTERNAL trigger mode (this is the default mode of operation, see ncCamSetTriggermode()). The line-rate should be multipled by the vertical binning (see ncCamGetBinningMode()) to recover the rate at which CCD rows are traversed on the sensor.

The rate at which lines are returned from the camera depends directly upon the exposure time requested (see ncCamSetExposureTime()) and upon the readout time (see ncCamGetReadoutTime()) which itself depends upon the readout mode and the layout of the image requested (e.g. width of ROIs, binning). These factors are taken into account in calculating the value reported by this function.

Note
Complete images are returned (see ncCamGetFramerateNominal()) according to the requested image height (see ncCamSetMRoiSize()).
Parameters
camThis parameter represents the camera structure used to acquire the images. This structure must have been previously defined using the ncCamOpen() function.
cameraRequestThis parameter is a flag that, when set to a non-zero value, will request up-to-date information directly from the camera. This process is not immediate, as communication through the serial link is required to retrieve the information. If this flag is set to 0, then the camera settings previously set within the application will be used. It is recommended to only set this flag to 1 when the ncCamSetExposureTime() function has not previously been used in the application or when there is a possibility that the camera settings have been changed using another application than the one currently running.
lineRate_kHzUpon return, this pointer will contain the nominal line-rate (in kHz) according to the camera settings in place.
Returns
This function returns "0" if successful and an error code if not.
int ncCamGetFramerateMaximum ( NcCam  cam,
int  cameraRequest,
double *  fps 
)

Reports the maximum number of frames that could be acquired per second, based on the readout setup on the camera.

Reports the maximum frame-rate that may be expected from the camera's current readout configuration. This is the nominal frame-rate (see ncCamGetFramerateNominal()) that can be achieved if both exposure and waiting time are set to zero so that the camera is reading out continuously, without pause; this would be achieved using the INTERNAL trigger mode (the default mode of operation, see ncCamSetTriggermode()).

Note
The rate at which frames are returned from the camera depends directly upon the exposure and waiting time requested (see ncCamSetExposureTime() and ncCamSetWaitingTime()) and upon the readout time (see ncCamGetReadoutTime()) which itself depends upon the readout mode and the layout of the image requested (e.g. width, height and position of ROIs).

Provides the time in milliseconds that the camera spends reading out image data after an exposure has ended.

The value reported by this function is the duration of the period (after each exposure, see ncCamGetExposureTime()) that the camera spends shifting accumulated charge for the purpose of reading out image data and sending it to the host computer. The readout time depends upon the readout mode and the layout of the image requested (e.g. width, height and position of ROIs).

Note
An image may not be returned after every period of exposure and subsequent readout time due to conditions such as frame latency (see ncCamGetFrameLatency()) or the use of TDI readout modes where the readout of individual lines may be buffered until a complete image has been read out.

int ncCamGetFrameTransferDuration ( NcCam  cam,

)

Provides the duration in milliseconds of the frame transfer portion of the readout time.

A camera with a sensor that uses frame transfer spends a certain portion of the readout time shifting the image from the exposure region to the storage region. During this time, any light received is not accumulated in the correct pixels. When using a camera at maximum frame-rate (i.e. no blanking time is set, see ncCamSetWaitingTime()) and especially when using ROI to accelerate readout this period constitutes a dead-time for pixels that would have received light. See ncCamGetIntegrationTime() for an exact measure of the effective exposure.

Note
If Fast Kinetics Mode is in use (see ncCamSetFastKineticsMode()), this function reports the time spent shifting the sub-image between consecutive exposures within a burst.

int ncCamGetFrameLatency ( NcCam  cam,
int *  frameLatency 
)

Retrieves the number of frames that must be returned before valid frames enter the processing or read queues.

Certain modes of operation, notably crop-mode Y (see also documentation for ncCamSetCropMode()), cause the camera to return frames taken previously after a certain number have flushed the acquisition pipeline. This is managed internally: initial frames are discarded and only the correct images will be returned. Consequently, after any period of blanking/waiting time (i.e. at the start of an acquisition or during a pause between triggers) the first frame available to either the processing or read queues will be delayed by the time required to return a single frame (see documentation for ncCamGetReadoutTime()) multiplied by the frame latency.

Note
A non-zero frame latency is incompatible with the use of non-zero blanking or waiting time (refer to documentation for ncCamSetWaitingTime()) and the trigger modes EXT_HIGH_LOW_EXP and EXT_LOW_HIGH_EXP which incur blanking/waiting time before each frame returned (refer to documentation for ncCamSetTriggerMode()). No images will be returned.
When using the trigger modes CONT_LOW_HIGH and CONT_HIGH_LOW, additional triggers must be supplied to compensate for the initial frames discarded in order to recover all of the frames expected. When using the trigger modes EXT_HIGH_LOW and EXT_HIGH_LOW, frame latency is handled internally and the correct number of images will be returned (see documentation for ncCamSetTriggerMode()).
Parameters
camThis parameter represents the camera structure used to acquire the images. This structure must have been previously defined using the ncCamOpen() function.
*frameLatencyThis parameter is a pointer indicating the number of frames that must be returned before valid frames enter the processing or read queues.
Returns
This function returns "0" if successful and an error code if not.
